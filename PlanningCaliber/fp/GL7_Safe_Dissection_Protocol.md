# GL7 安全解剖プロトコル (Safe Dissection Protocol)

- 対象: `structural/governance_pipeline.py` (execution_governance)
- 作成日: 2026-07-01
- 前提文書: `KN-004_Registry_Session_LanguageRecord.md`, `MoCKA_KN004_System_Map.md`
- 実行環境: くろこ側セッション(実ファイルシステムアクセス必須)

---

## 0. 大原則

**この作業は「読む」であって「直す」ではない。**

MoCKA IMMUTABLEルールに基づき、本プロトコル実行中は以下を厳守する:

- コードの書き換えは一切行わない(コメント追加・整形も含む)
- 「ここを直すべき」という発見があっても、その場で修正しない
- 発見はすべて本プロトコルの記録フォーマット(セクション5)に記述し、
  修正判断は別セッション・別PHL記録として切り離す
- 読解の過程で実行(`python governance_pipeline.py`等)が必要な場合も、
  副作用(ファイル書き込み・状態変更・abort発火)を伴わない形
  (dry-run、静的読解、テスト環境限定)に留める

理由: GL7はガバナンス層の本体であり、他ファイルより一段慎重に
扱うべき対象である。ここで生じた変更は「ガバナンスを変更する」
という意味を持ち、通常のバグ修正とは重みが異なる。

---

## 1. 読解の目的(スコープ限定)

今回のGL7解析で確定させるべき問いは、以下の3つに限定する。
スコープを広げない(全機能の完全理解を今回の目標にしない)。

1. **abort条件は何か** — どのような状態でGL7は実行を止めるのか
2. **abortはいつ発火するのか** — 発火のトリガー・タイミング
3. **GL7は他レイヤ(registry_store/MCP)とどう接続しているか**
   (`MoCKA_KN004_System_Map.md` セクション6の空白を埋める)

これ以外の発見(コード品質、リファクタ候補等)は「その他の観測」として
記録はするが、今回の分析目的には含めない。

---

## 2. 読解順序(推奨ステップ)

### STEP 0: 既存Dry Runログの実体確認(2026-07-01の観測を受けて追加)

**背景(事実のみ)**: `data/lever_essence.json` のOPERATIONログに、
2026-07-01付で「KN-004 Registry統合テスト」「GL7 Dry Run通過(abortなし)」
「`ID_TEST_001`」「`source_record=PHL_TEST_001`」という記述が存在することが
確認された。同ログは記述の途中で次のエントリに切り替わっており、
`PHL_TEST_001`に対応する実イベント記録の有無は本セッションでは
確認できなかった(ログ自体が途切れているため)。

**この背景から導ける事実はここまでで、それ以上の解釈(構造仮説等)は
本ステップの対象外とする。** 解釈はSTEP 0の結果が出てから、必要であれば
別途行う。

**確認事項(このステップで行うこと)**:
1. `data/lever_essence.json` の該当ログを完全な形で読む
   (ログが本当に途中で切れているのか、単に本セッションでの取得範囲が
   途中までだったのかを切り分ける)

   ```bash
   grep -n "PHL_TEST_001\|ID_TEST_001\|GL7 Dry Run" data/lever_essence.json
   # 該当行の前後を広めに表示して、ログが物理的に途切れているか、
   # JSON構造上の別要素に続いているだけかを確認する
   ```

2. `PHL_TEST_001` に対応する実イベント記録を、PHLのイベントストア
   (`E20260701_...`形式のもの)から検索する。見つかれば「記録漏れ」
   ではなかったことになり、見つからなければ「記録欠落」が事実として
   確定する

   ```bash
   grep -rn "PHL_TEST_001" --include="*.json" .
   grep -rln "PHL_TEST_001" .  # ファイル横断で存在有無のみ先に確認
   ```

3. `ID_TEST_001` が実際にregistryに存在するか(`mocka_registry_get`等で)
   確認する

   ```bash
   grep -rn "ID_TEST_001" PlanningCaliber/workshop/
   # MCPツールが使える場合は mocka_registry_get(layer=identity, id=ID_TEST_001) 等
   # で直接照会する方が確実
   ```

4. 上記1〜3の結果を、通常のGL7読解記録フォーマット(セクション5)とは
   別に、以下の簡易フォーマットで記録する:

```markdown
### STEP 0 結果 — 既存Dry Runログの実体確認

- ログ全文: [取得できた場合はここに引用/要約]
- ログが途切れている原因: [判明した場合のみ記述、不明なら「不明」と明記]
- PHL_TEST_001実イベント: [発見/未発見]
- ID_TEST_001のregistry実在: [確認/未確認]
- この時点での解釈: [事実の列挙のみ。「二重構造」等の構造仮説は
  ここでは立てない。仮説を立てる場合は別文書として分離する]
```

STEP 0完了後、その結果を踏まえてSTEP 1(コード解析)に進む。
STEP 0の結果が「単純な記録漏れ」であれば、GL7読解の優先度・観点に
大きな変更は不要。「記録欠落が構造的」と判断できる材料が揃った場合のみ、
grounding系統の読解時にこの論点を重点的に扱う。

## 3. abort条件 3系統(分類の軸)

読解時、発見した条件は以下3系統のいずれかに分類する。
どれにも当てはまらない場合は「未分類」として保留し、無理に当てはめない。

| 系統 | 定義 | 着目点 |
|---|---|---|
| **grounding** | 実行の前提となる事実確認(ファイル存在、状態整合性等)が取れているかの検証 | 「何を確認してからでないと進まないか」 |
| **filesystem change** | ファイルシステムへの書き込み・変更操作に対する制御 | 「どんな書き込みを許可/拒否するか」 |
| **action validation** | 実行しようとしているアクション自体の妥当性検証 | 「何をしようとしているアクションか、それは許可された種類か」 |

---

## 4. 完了条件(いつ「読み終わった」と判断するか)

以下がすべて満たされた時点で、本プロトコルの解析フェーズを完了とする。

- [ ] 3系統それぞれについて、該当コード箇所が最低1つ以上特定されている
      (無い場合は「その系統の実装は本ファイル内に存在しない」と明記)
- [ ] GL7と他レイヤの接続有無(import/呼び出し関係)が確認されている
- [ ] セクション5のフォーマットで記録が作成されている
- [ ] コードへの変更が一切行われていないことが`git status`で確認されている
