# MoCKA思想進化史 v0.1（現時点の復元結果・Revision 0）

位置づけ：憲法でもADRでも実装仕様でもない。「なぜその制度が生まれたのか」を説明する設計思想史であり、同時に今後の設計判断の基準書でもある。今後の資料・議事録により更新され得る。

## 第0世代｜現場
プラント制御・設計・現場・経営・改善という30年の実務経験。「なぜ同じ説明を何度もするんだ」「なぜ記録が残らないんだ」「なぜ前回を活かせないんだ」という問いが原点。MoCKAの出自はソフトウェアではなく現場にある。

## 第1世代｜AIの根本欠点の発見
AIは毎回ゼロから考え直す。考えたことが経験として残らない。同じ説明、同じ議論、同じ設計、同じ失敗が繰り返される。

## 第2世代｜経験を残す制度が必要
欠点：考えたことが消える。
第一実装：Orchestra（経験の一次保存層）。チャット保存ツールではなく、経験を失わないための最初の制度。
しかし、保存だけでは次に使えないという欠点が残る。

## 第3世代｜経験を継承する制度が必要
欠点：保存した経験の山から、何が重要か分からず使えない。
実装：Relay（経験継承制度）。圧縮技術ではなく、経験を次のAIへ継承するための制度。
しかし、Relayが継承するのは「何を話したか」であり、「なぜそう考えたか」ではないという欠点が残る。

## 第4世代｜理由を忘れない制度が必要
欠点：AIは理由を忘れる。文脈を渡されても判断の背景は薄れる。
構想：PHL（再注入装置。保存装置ではない）。

## 第5世代｜AIの能力ではなく制度で品質を保証する
欠点：理由を持っていても、AIはそれを守らない。
原則：AIの能力ではなく、制度によって品質を保証する（運用原則：「AIを信じるな、システムで縛れ」）。
実装：PHI-OS / event_gate / Trust Boundary（制度実行層）。

## 第6世代｜Reasonを保存する萌芽
欠点：制度はイベントを記録するが、判断の理由そのものは点としてしか残らない。
萌芽：judgement_reasonテーブル。当時は名前のない実験的な実装だったが、現在の観点から見ると、Reason Capture思想へ繋がる最初期の萌芽として再解釈できる（断定はしない）。

## 第7世代｜理由そのものを知識として扱う
構想：Reason Capture。散らばった理由を横断的に検索・参照可能な知識単位として扱う。

## 第8世代｜理由が制度になる
構想：Institutional Reasoning。Reasonが個別の記録から制度として機能する段階へ。

## 第9世代｜知識は活性化されて行動を変える（North Star）
Reason → Knowledge → Institution → Behavior。
Knowledgeは保存されるだけでは終わらない。AIの行動を変えるために活性化される（Knowledge Activation）。この一文によってPHI・Memory・Relay・Prismが一本の系譜として繋がる。

## 設計原則
MoCKAは、新しい機能を追加するために進化したのではない。AIが持つ一つの欠点を発見し、その欠点を制度によって補うことを繰り返して進化してきた。したがって、MoCKAの各コンポーネントは独立した製品ではなく、AIの欠点を制度で補完するための進化の系譜として理解される。

## 本文書の位置づけ（基準書としての機能）
今後何か新しい機能や制度を考えるときは、次の3つを説明できなければMoCKAの思想と整合しない。
①AIのどんな欠点を解決するのか
②その欠点に対する制度は何か
③その制度はKnowledge → Institution → Behaviorの流れにどう組み込まれるか
