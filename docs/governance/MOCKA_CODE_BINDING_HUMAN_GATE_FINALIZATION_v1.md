## 1. 承認単位

選択：段階承認を採用する。

裁定理由：

* MoCKAは段階的検証を前提として構築されている。
* 設計フェーズにおいても、Phaseごとに監査・固定・再検証を繰り返してきた。
* Code Bindingについても同様に、実装単位ごとにHuman Gateを通過させることで、変更影響範囲を限定できる。
* 問題発生時の原因特定およびロールバックが容易になる。

---

## 2. Rollback Baseline（Rollback Policy）

選択：固定タグへの一律ロールバックではなく、問題発生地点を起点とした原因追跡型ロールバックを採用する。

裁定理由：

* MoCKAの目的は単なる復旧ではなく、原因の把握と再発防止にある。
* 問題が発生した実装単位および関連変更範囲を監査し、影響範囲を確認した上で復旧を行う。
* 必要に応じて既存タグおよびコミットを参照するが、ロールバックそのものは監査結果に基づいて決定する。

原則（遵守順序）：

1. 問題箇所の特定
2. 周辺変更の監査
3. 影響範囲の確認
4. 復旧方針決定
5. 再検証

---

## 3. Completion Criteria

選択：実際に稼働可能であり、再現可能であり、監査可能な状態を完了と定義する。

裁定理由（完了条件）：

1. 実装対象コードが正常に動作すること
2. 既存テストが全て通過すること
3. 新規テストが全て通過すること
4. 再検証を実施し同一結果が得られること
5. Human Gate監査を通過すること
6. 関連仕様書およびMarkdown文書が整備されていること
7. 実装内容と文書内容が整合していること

完了の定義は「コードが存在すること」ではなく、「運用可能状態に到達していること」とする。

---

## 4. Audit Artifact

選択：成果物単体ではなく、関連性を含めて記録する。

裁定理由（記録対象）：

* 実装内容
* 仕様書
* 設計文書
* テスト結果
* 監査結果
* 関連コミット
* 関連タグ
* 依存関係

原則：各成果物は単独で保存するのではなく、

* なぜ作られたか
* どの仕様に基づくか
* 何を変更したか
* どの成果物と関係するか

を追跡可能な状態で記録する。MoCKAにおける監査成果物は、単なる保存物ではなく、設計・実装・検証を接続する証跡として扱う。
