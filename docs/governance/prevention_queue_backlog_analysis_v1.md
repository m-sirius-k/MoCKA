## 2. 整合性チェック（重複・既解決事象の確認）

### 2.1 mcp_schema_hash（1,472件、82%）— 内容の内訳

- 「ハッシュ変更検出!」（前回/今回ハッシュの提示のみ）: 約946件
- 「例外: Expecting value: line 1 column 1 (char 0)」（JSON読み込み失敗）: 513件

「ハッシュ変更検出」は、`mocka_mcp_server.py`を編集する度に`mcp_schema_hash.json`の
ハッシュ値が変わることそのものを検知しているだけであり、これは.claude/CLAUDE.mdの
「mocka_mcp_server.py 変更後の必須手順」で**正規の運用として明文化されている変更**
である（TODO_385のStep1作業でも実際にこの手順を踏んだ）。つまり**ほぼ全件が
「正常な開発活動を異常として検知した」誤検知（NOISE）**と判断できる。

「例外: Expecting value」は、ハッシュファイル書き込み中（一時的に空/不完全な状態）に
監視スクリプトが読み込みを試みたタイミング競合によるものと推定され、これも実害のない
一時的な読み取りエラーである可能性が高い（断定はしない。watcherのリトライ設計が
無い可能性があり、watcher側の改善要否は別途検討事項）。

### 2.2 relay_dom_selector（211件、12%）

170件が「Relay未接続: last_ping なし（拡張が起動していないか /relay/ping 未実装）」。
これはRelay拡張機能が単に起動していない状態を検知しているだけで、コード上の不具合
ではなく**環境状態（拡張未起動）の検知**である可能性が高い。28件の「HTTP Error 403」
・6件の「MoCKAサーバー未起動」も同様に一時的な接続状態の検知と考えられる。

6件のみ「HTTP 200, selector=MISSING」という、DOM構造側の問題を示唆する内容があった。
日時はすべて2026-06-01〜02で、TODO_374調査時に発見した別件のDOM selector drift
インシデント（2026-06-26/27、`E20260627_753188422ec60`）とは**時期が異なり直接の
対応関係は確認できなかった**（断定せず、念のため個別確認の対象に残す）。

### 2.3 phi_os_audit（54件、3%）— TODO_347との関係（重要・要注意）

内容は「Direct Write違反N件 | Gate経由率X%(<95%)」という、PHI-OS Gateの経由率を
監視するアラート。これは**TODO_347「PHI-OS Gate経由率64.4%の残課題対応」
（2026-06-25完了）と同じ指標を対象にしている**。

TODO_347の note によれば、2026-06-25にgate_policy.pyへ集計ロジックを統一する修正が
行われ、本物のエンドポイント実測で「gate_passthrough_rate_percent=82.96%→100.0%」
まで改善したと記録されている。

**しかし、本バックログのphi_os_audit 54件の発生日付は2026-06-18〜2026-06-28に
分布しており、TODO_347完了日(06-25)以降の日付（06-26〜06-28）にも複数件存在する**。
これは以下のいずれかを意味する可能性がある（断定しない）。

- (a) prevention_queueへの投入元（health_check.py等）がTODO_347で統一された
  `gate_policy.py`の新ロジックをまだ参照しておらず、旧い・低い数値（2-3%台、
  TODO_347が言及した64.4%とも、修正後の100%とも異なる）を引き続き算出している
  可能性がある（**TODO_347の修正が一部経路に反映されていない疑い**）。
- (b) 全く別の集計ロジック・別の閾値判定がhealth_check.py内に存在し、TODO_347の
  対象（dashboard.py/event_gate.py/health_check.pyの3重複実装の統一）に
  含まれていなかった可能性。

このグループは**「対応不要」と断定できない**。TODO_347が解決したはずの指標と
矛盾する値が、修正完了日より後も生成され続けている点は、別途確認が必要な懸念点
