## 2. Observer / Advisorとの境界再定義

### Observerとの違い（事実 vs 兆候）

```
Observerは「見る・説明する」のみであり、解釈を加えない状態説明
（Phase10-1第2章）に限定される。本主体は「パターンの逸脱・
残差」という解釈的検出を行う点で質的に異なる。

ただし、この「解釈の有無」という境界軸は、Phase10-3論点1
Option C（意味的推論主体）でも同型の論点として既に記録済み
（PHASE10_3_REASONING_DEFINITION_COMPARISON_v1.md「Observerとの
違い: 中程度」）。同じ構造的論点が本主体にも再発生する。
```

### Advisorとの違い（提案 vs 違和感検出）

```
Advisorは「見る・説明する・提案する」が採択権は持たず、役割の
1つに「矛盾点の指摘」（Phase10-2第3章役割4）が既に含まれる。
これは「違和感検出」と機能的にほぼ同義であり、既存Advisor契約
の中に本主体の機能の一部が既に包含されている可能性が高い。

重複領域: 「矛盾点の指摘」（Advisor既存機能）と「違和感検出」
（本主体の中核機能）の境界が、5名称候補のいずれを選んでも
未確定のまま残る最大の論点。
```

### 重複領域排除の試み（観察のみ、採用しない）

```
本文書で識別できた境界案は以下2案（いずれも採用ではない、
観察として記録）:

案1: 入力ソースによる境界
    Advisorはcandidate群（Projection層出力等、既に構造化済みの
    候補）を扱うのに対し、本主体は生ログ・履歴・event stream
    （未構造化の時系列データ）を直接扱う、という区分。

案2: 起動形態による境界
    Advisorは要求応答型（Advisor Request Contract、Phase10-3
    advisor_adapter_contract_v1.md）であるのに対し、本主体は
    常時監視型（Phase7-C Drift Monitorのobserve()ループと同型）
    である、という区分。

いずれの案も既存資料からの再構成であり、本文書はどちらか一方を
採用しない。
```

## 4. 制度位置の確定（構造候補の提示のみ、採用ではない）

### 4層モデルとの関係（観察）

```
博士提示の4層モデル: Observer=事実/Advisor=提案/Signal系=兆候/
Human Gate=判断。

これは既存Phase10-0の3 Node分類（Advisor/Observer/Reasoning）に
対し、「Reasoning」を置き換えるのではなく、4番目の独立層として
追加する案として読める（本文書ではこの解釈の正否を判定しない）。

もしSignal系がReasoningそのものを置き換える場合、
PHASE10_3_DECISION_DEPENDENCY_MAP_v2.mdが確定した依存構造
（論点1 Reasoning Definitionが論点2 Candidate Authority・論点3
Projection Boundaryの上位論点である構造）の前提自体が変わる
可能性がある。この影響範囲は本文書では評価しない（未解決事項）。
```

### Human Gateとの関係（介入有無、再利用可能な既存パターン）

```
Phase7-C Drift Monitorが既に確立しているHumanGateHookパターン
（notify()空実装デフォルト、自動修復恒久禁止）は、本主体にも
そのまま再利用できる可能性が高い（既存資料の再解釈のみで
導出可能）。

この「記録・通知のみ、自動修復なし」という型は、既存5契約系列
（Phase7-B5/B6/B7、Phase9-3A、Phase10-0）が一貫して採用している
パターンであり、本主体がいずれの名称候補を採るとしても、この型
からの逸脱は既存制度との整合性を損なう可能性が高い（観察、
結論ではない）。
```
