# Phase10-3 Collision Gap Analysis v1

Status: ANALYSIS ONLY（分析のみ。結論禁止・契約作成禁止）
Date: 2026-06-23

本文書はPHASE10_3_COLLISION_AMPLIFICATION_AUDIT_v1.mdで確定した
「collision増幅」の制度的空白について、なぜ空白が生じたか・
どの契約系列に存在しないか・Reasoning契約に入れる場合と
入れない場合の影響を整理する。結論は出さない。

## なぜ空白が生じたか（既存契約の設計時系列からの整理）

```
Phase7-B5（Collision Governance）の設計時点:
    collisionは「既存データ（decision_trace.json/merge_graph.json）
    間の構造的不一致から検出される対象」として位置づけられた
    （PHASE10_3_COLLISION_GOVERNANCE_STUDY_v1.md確認元1）。
    この時点でcollisionは「発見されるもの」であり「生成・増幅
    されるもの」ではなかった。そのため、解消（resolve）の禁止は
    設計の中心に置かれたが、生成・増幅という能動的局面は設計
    対象に含まれなかった。

Phase9-1/9-3A（Projection層）の設計時点:
    Projection層は「複数候補を同時に生成する」設計
    （ProjectionResult.candidatesは常にSequence）を採用したが、
    これは「単一解収束の禁止」という防御的な目的のための構造
    であり、「複数候補がcollisionを生む」という事象そのものを
    主題として扱っていない。Collision Policy
    （「候補間競合は解消しない」）はPhase7-B5の継承であり、
    Phase7-B5と同様に解消の禁止が中心に置かれた。

Phase10-1/10-2（Observer/Advisor Node）の設計時点:
    Observer/Advisorはいずれも候補生成権限を持たないため
    （Phase10-1/10-2第2章で候補生成系操作は禁止済み）、
    「生成によるcollision増幅」という事象自体がそもそも発生
    しない設計になっている。したがってこの2契約に増幅の規定が
    無いのは、規定漏れではなく、増幅が発生する条件
    （candidate生成権限）がそもそも存在しないことの論理的帰結
    である。
```

```
空白が生じた構造的理由（整理）:
    既存4契約系列はいずれも「collisionは外から与えられる、また
    生まれない」という前提のもとで設計されており、「Nodeの
    能動的な生成行為がcollisionを増やす」という事象を想定する
    必要が無かった。Phase10-1/10-2は候補生成権限自体を禁止する
    ことでこの前提を維持したが、Reasoning Nodeが候補生成権限
    （論点2 Candidate Authority）を持つ可能性が初めて生じたため、
    この空白が初めて実際の問題として表面化した。
```

## どの契約系列に存在しないか（再整理、PHASE10_3_COLLISION_AMPLIFICATION_AUDIT_v1.mdの引用）

```
Phase7-B5:        増幅の明記なし（解消の明記のみ）
Phase9-1/9-3A:    増幅の明記なし（Collision Policyは解消禁止のみ、
                  Ranking Policyは収束方向の制約のみ）
Phase10-1:        増幅の明記なし（候補生成自体が禁止のため
                  発生条件が無い）
Phase10-2:        増幅の明記なし（同上）

結果: 4契約系列すべてに増幅の明記が存在しない。
```

## Reasoning契約に増幅規定を入れる場合の影響

```
影響1: 既存4契約系列との非対称性が生じる。
    Phase7-B5/Phase9層/Phase10-1/10-2のいずれも増幅を明記して
    いないため、Reasoning Node契約のみが増幅について明文の
    規定を持つことになる。これは「Reasoning Nodeが候補生成
    権限を持つ唯一のNode種別である（候補2のModel選択次第）」
    という事実に対応した特別規定として位置づけられる可能性が
    あるが、既存制度の一貫性（条文密度の差）という観点では
    異質になる。

影響2: 「禁止」を入れる場合、Candidate Authority（論点2）の
    Model選択に制約が生じる可能性がある。
    例えば「collision増幅を禁止する」という規定を置く場合、
    Model D（reconstruct含む、collisionリスク最高、PHASE10_3_
    LEAST_AUTHORITY_AUDIT_v1.md参照）のような操作が事実上
    制限される、または運用上の追加チェックが必要になる可能性
    がある（本文書ではこの制約の強さを判定しない）。

影響3: 「許可」を入れる場合、既存のPhase7-B5絶対禁止4項目
    （自動マージ等、collisionを減らす方向の禁止）との関係を
    どう整理するかという論点が新たに生じる。「増やすことは
    許可するが減らす操作は禁止する」という非対称な構造が
    既存制度の一貫性とどう整合するかが問われる。
```

## Reasoning契約に増幅規定を入れない場合の影響

```
影響1: 既存4契約系列との形式的整合性は保たれる。
    increaseに関する規定が無いという状態が、既存契約と
    同型のまま維持される。

影響2: 運用上の判断が個別事例に委ねられる。
    Reasoning Nodeが実際に候補生成権限を行使した結果、
    collisionが増幅する事例が発生した場合、それを許容するか
    問題視するかの基準が契約上存在しないため、Human Gateが
    個別に判断する負荷が生じる（PHASE10_3_HUMAN_GATE_LOAD_
    ANALYSIS_v1.mdで別途分析）。

影響3: 将来的な制度的空白の累積。
    Reasoning Node契約でも増幅を扱わない場合、「解消のみ
    規定され生成・維持・増幅は規定されない」という既存パターン
    がそのまま継続される。これは既存制度の設計思想
    （Phase7-B5以来の「衝突を消さない」という防御的境界）との
    一貫性は保たれるが、空白自体は解消されない。
```

## 結論

禁止/許可）についても結論を出していない。次工程（Step6
